# <span style="color:green">欢迎来到逆向的第一篇文章</span>

ps:由于我也不是很会逆向,我也只是边学边写,谈谈自己的想法(不喜勿喷)😋

在很多人的印象中，**逆向工程**就是：

- 打开 IDA
- 把文件扔进去
- 找 flag

但在我真正开始接触逆向之后，我发现这完全不是本质。

让我们从一个更朴素的问题开始看看：

> CPU 到底是如何一步一步执行这个程序的？

为了说明这一点，我们从一个几乎不能再简单的程序开始。🤭

```c
#include <stdio.h>

int add(int a, int b) {
    return a + b;
}

int main() {
    int x = 3;
    int y = 4;
    int z = add(x, y);
    printf("%d\n", z);
    return 0;
}
```

让我们从汇编视角看看这样一个简单的函数调用小程序到底干了哪些事情

## 函数的入口

```asm
push rbp
mov rbp, rsp
sub rsp, 0x20
```

这三条指令意味着一件事:

一个函数开始了

当前函数正在建立自己的栈帧

在逆向中,只要看到这些,你就可以确定:这是一个函数的入口。

## 局部变量是怎么出现的？

源码中有：

```c
int x = 3;
int y = 4;
```

在汇编里，它们通常表现为：

```asm
mov DWORD PTR [rbp-0x4], 0x3
mov DWORD PTR [rbp-0x8], 0x4
```

rbp 是当前函数**栈帧**的基址

rbp-0x4、rbp-0x8 是局部变量

常量 3、4 被直接写进栈内存

因此

> 局部变量 ≈ 栈上的一块内存

逆向不是猜测变量名是什么

而是判断：

> 这块内存什么时候被写?什么时候被读?

## 函数调用与传参

```c
z = add(x, y);
```

在 x86-64 Linux 下，常见的汇编形式是:

```asm
mov eax, DWORD PTR [rbp-0x4]
mov edi, eax
mov eax, DWORD PTR [rbp-0x8]
mov esi, eax
call add
```

关键:

参数被送进**寄存器**

call 指令跳转到函数

返回值会放在 eax / rax

## 函数中发生了什么?

add 函数的核心汇编通常非常短：

```asm
push rbp
mov rbp, rsp
mov eax, edi
add eax, esi
pop rbp
ret
```

edi放着第一个参数 a

esi放着第二个参数 b

而eax是返回值寄存器

这条指令：

add eax, esi

就对应着源码里的：

return a + b;

到这里你应该意识到一件事：

C 语言的 return，本质上只是把值放进 eax!

## <span style="color:blue">pwn与逆向的联系</span>

```asm
call add
...
ret
```

call 会：

把返回地址压栈

跳转到目标函数

ret 会：

从栈中取出返回地址

跳回调用点

因此

覆盖返回地址 = 控制程序执行流

栈溢出能 hijack 控制流

> pwn 和逆向,在这里,产生了交集!

## printf从何而来?

```asm
call printf@plt
```

printf 不在当前程序里

它来自 libc 动态链接库

这里只是通过 PLT 进行一次跳转

逆向中看到 xxx@plt，你应该立刻反应：

> 这是外部函数调用，不是程序逻辑本身!

## **总结**

逆向不是从做题开始的，

而是从能完整看懂执行流开始的。

当你能用汇编解释：

一个函数怎么被调用

参数从哪里来

返回值往哪里走

> 那你已经迈进逆向的大门了!
